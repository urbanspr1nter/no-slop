Mastering CUDA

A transition guide for C programmers moving from Sequential to Massive Parallelism.

1. What is CUDA?

If you are coming from a standard C background on Linux, you are likely used to the way a CPU (Central Processing Unit) operates. A CPU is a highly sophisticated "generalist." It is designed to execute a sequence of complex instructions very quickly, handling branching logic, system interrupts, and complex memory management with ease. In a typical C program, you write a sequence of steps: first do A, then do B, then loop through C. This is sequential execution.

CUDA (Compute Unified Device Architecture) is a parallel computing platform and programming model created by NVIDIA. It allows you to offload massive amounts of simple, repetitive mathematical tasks to the GPU (Graphics Processing Unit).

Think of the difference this way: A CPU is like a single, incredibly fast mathematician who can solve any equation you give them, one after another. A GPU is like an entire stadium filled with thousands of high-school students. Individually, they aren't as smart or fast as the mathematician, but if you give them all the same simple addition problem (e.g., "add these two numbers"), they can solve ten thousand problems in the time it takes the mathematician to solve ten.

In CUDA, we stop thinking about "How do I loop through this array?" and start thinking about "How do I assign one element of this array to every available worker?"

2. How to Compile a CUDA Program

In the Linux world, you are probably used to `gcc` or `clang`. For CUDA, you will use a specialized compiler called nvcc (NVIDIA CUDA Compiler).

CUDA files typically use the .cu extension. Unlike a standard .c file, a .cu file contains two types of code:

  1. Host Code: Standard C/C++ code that runs on your CPU.
  2. Device Code: Special "kernel" functions that are compiled to run on the GPU.

When you run nvcc, it acts as a wrapper. It splits your code: it sends the CPU parts to a standard host compiler (like gcc) and handles the GPU parts using its own internal compiler technology.

Terminal Command

nvcc my_program.cu -o my_program

To ensure you have everything ready, you'll need the NVIDIA Driver and the CUDA Toolkit installed on your Linux machine. You can verify your installation by running nvidia-smi in your terminal.

3. Writing Your First CUDA Program

The "Hello World" of parallel computing is Vector Addition. Given two arrays, A and B, we want to create a third array C where each element is C[i] = A[i] + B[i].

In C, you would use a for loop. In CUDA, we write a Kernel.

vector_add.cu
#include <stdio.h>
#include <cuda_runtime.h>

// This is the KERNEL. 
// The __global__ keyword tells the compiler this runs on the GPU 
// and is called from the CPU.
__global__ void vectorAdd(const float *A, const float *B, float *C, int N) {
    // Calculate the unique index for this specific thread
    int i = blockDim.x * blockIdx.x + threadIdx.x;

    // Boundary check: ensure we don't access memory out of bounds
    if (i < N) {
        C[i] = A[i] + B[i];
    }
}

int main() {
    int N = 1 << 20; // 1,048,576 elements
    size_t size = N * sizeof(float);

    // 1. Allocate memory on the HOST (CPU)
    float *h_A = (float *)malloc(size);
    float *h_B = (float *)malloc(size);
    float *h_C = (float *)malloc(size);

    // Initialize arrays
    for (int i = 0; i < N; i++) {
        h_A[i] = 1.0f;
        h_B[i] = 2.0f;
    }

    // 2. Allocate memory on the DEVICE (GPU)
    float *d_A, *d_B, *d_C;
    cudaMalloc(&d_A, size);
    cudaMalloc(&d_B, size);
    cudaMalloc(&d_C, size);

    // 3. Copy data from Host to Device
    cudaMemcpy(d_A, h_A, size, cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_B, size, cudaMemcpyHostToDevice);

    // 4. Launch the Kernel
    // We define how many threads per block and how many blocks.
    int threadsPerBlock = 256;
    int blocksPerGrid = (N + threadsPerBlock - 1) / threadsPerBlock;

    vectorAdd <<<<< blocksPerGrid, threadsPerBlock >>>>> (d_A, d_B, d_C, N);

    // 5. Copy result back from Device to Host
    cudaMemcpy(h_C, d_C, size, cudaMemcpyDeviceToHost);

    // Verify
    printf("Result at index 0: %f (Expected 3.0)\n", h_C[0]);
    printf("Result at index N-1: %f (Expected 3.0)\n", h_C[N-1]);

    // 6. Cleanup
    cudaFree(d_A);
    cudaFree(d_B);
    cudaFree(d_C);
    free(h_A);
    free(h_B);
    free(h_C);

    return 0;
}

Notice the pattern: Allocate Host memory $\rightarrow$ Allocate Device memory $\rightarrow$ Copy data to Device $\rightarrow$ Launch Kernel $\rightarrow$ Copy results back $\rightarrow$ Cleanup. This "Copy-Compute-Copy" cycle is fundamental to CUDA.

4. The Execution Model: Grids, Blocks, and Threads

In C, you control a single execution pointer. In CUDA, you control a hierarchy of parallel workers. To understand how to map your problem to the GPU, you must understand the Grid $\rightarrow$ Block $\rightarrow$ Thread hierarchy.

graph TD subgraph Grid ["Grid (The entire problem)"] subgraph Block1 ["Block 0"] T1["Thread 0"] T2["Thread 1"] T3["Thread 2"] T4["Thread 3"] end subgraph Block2 ["Block 1"] T5["Thread 0"] T6["Thread 1"] T7["Thread 2"] T8["Thread 3"] end
  • Thread: The smallest unit of execution. Each thread executes the kernel function.
  • Block: A group of threads that can communicate with each other through fast Shared Memory.
  • Grid: A collection of blocks that together represent the entire parallel operation.

In the code example above, we used blockDim.x, blockIdx.x, and threadIdx.x. These are built-in variables that allow a thread to "know" who it is. The magic formula int i = blockDim.x * blockIdx.x + threadIdx.x; is how a thread calculates its global position in the large array.

5. Success Strategies: The CUDA Mental Model

To be successful with CUDA, you cannot simply "port" your C code. You must re-architect your thinking. Here are the three pillars of the CUDA mindset:

Data Locality

Moving data between CPU and GPU is expensive. Minimize transfers. Keep data on the GPU as long as possible.

Coalesced Access

Threads in a block should access adjacent memory addresses. This allows the hardware to group memory requests, making it much faster.

Avoid Divergence

Avoid complex if/else logic inside kernels. If threads in a block take different paths, the hardware has to run them sequentially (Divergence).

6. Beginning Project Ideas

Once you have mastered vector addition, don't stop. Try these projects in order of difficulty:

  1. Matrix Multiplication: The classic. It teaches you about 2D grids and shared memory.
  2. Image Grayscale/Blur: Treat an image as a 2D array of pixels. It's a great way to visualize 2D thread mapping.
  3. N-Body Simulation: Simulate gravity between particles. This will push you to think about $O(N^2)$ complexity and optimization.
