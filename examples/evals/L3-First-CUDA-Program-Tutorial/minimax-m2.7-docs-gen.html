🚀 What is CUDA?

CUDA stands for Compute Unified Device Architecture. It is NVIDIA's parallel computing platform and programming model that enables dramatic increases in computing performance by harnessing the power of the graphics processing unit (GPU). If you know C programming, CUDA is essentially an extension of C that lets you write code capable of executing thousands of threads simultaneously on a GPU.

The Fundamental Difference: CPU vs GPU

Before diving into CUDA, you need to understand why GPUs are so powerful for certain types of computations. Think about the difference between a Formula 1 race car and a pickup truck:

🔲

CPU

Few cores optimized for sequential serial tasks. Great at complex logic, branching, and running operating systems.

🔳

GPU

Thousands of smaller cores optimized for parallel tasks. Excels at performing the same operation on many data elements simultaneously.

Your typical C program runs entirely on the CPU. The CPU has maybe 8, 12, or 32 cores (if you're lucky), and each core executes one thread at a time. A modern GPU can have thousands of cores, each capable of running its own thread. This is why GPUs are so powerful for tasks that can be parallelized.

How CUDA Extends C

CUDA adds several key features to the C language that allow you to write code for the GPU:

C Feature CUDA Addition Purpose
Functions __global__, __device__ Define functions that run on GPU
Pointers cudaMalloc(), cudaMemcpy() Allocate memory on GPU
Function calls Kernel launch syntax <<<...>>> Launch parallel computation
Variables threadIdx, blockIdx Identify parallel threads

The good news is that if you know C, you already know most of the syntax. CUDA is not a completely new language—it is C with extensions. The tricky part is learning to think in parallel, which we'll cover throughout this tutorial.

What Can You Accelerate?

CUDA excels at data-parallel computations—problems where the same operation needs to be performed on many different pieces of data independently. Here are some classic examples:

Scientific Computing

  • Matrix operations and linear algebra
  • Monte Carlo simulations
  • Finite difference methods
  • Molecular dynamics

Emerging Applications

  • Deep learning and neural networks
  • Image and video processing
  • Computer graphics and ray tracing
  • Cryptocurrency mining (historically)

💡 Key Insight

The golden rule: if your nested for loops iterate over large arrays and each iteration is independent of the others, your problem is likely a great candidate for GPU acceleration.

⚙️ How to Compile a CUDA Program

Compiling CUDA programs on Linux requires the NVIDIA CUDA Toolkit, which includes the nvcc compiler driver. This section covers everything you need to know about getting your CUDA code compiled and running.

Prerequisites

Before you can compile CUDA programs, you need:

  1. An NVIDIA GPU — Check with lspci | grep -i nvidia or nvidia-smi
  2. NVIDIA Driver — Install via your distribution's package manager
  3. CUDA Toolkit — Download from NVIDIA's website

⚠️ Important

Make sure your driver version supports the CUDA Toolkit version you install. You can check your driver version with nvidia-smi—the top right shows the supported CUDA version.

The nvcc Compiler

The NVIDIA CUDA Compiler (nvcc) is your gateway to compiling CUDA programs. It handles both host (CPU) code and device (GPU) code compilation. The basic syntax should feel familiar if you've used gcc:

Basic compilation command 
nvcc my_program.cu -o my_program

CUDA source files typically use the .cu extension, though .cuh is used for header files. The compiler will produce a standard executable that you can run just like any other Linux program.

Common Compilation Flags

Flag Example Description
-o -o output_name Specify output executable name
-arch -arch=sm_70 GPU architecture (e.g., sm_70 for Volta)
-O -O3 Optimization level (0-3)
-g -g Generate debug symbols
-lineinfo -lineinfo Line-level profiling information
-Xcompiler -Xcompiler -fPIC Pass options to the host compiler

Finding Your GPU's Compute Capability

Different NVIDIA GPUs have different architectures. When compiling, you should target your specific GPU. You can find your GPU's compute capability using nvidia-smi or by looking it up:

Check GPU information 
nvidia-smi
# Or for detailed GPU info:
nvidia-smi -L

💡 Common Compute Capabilities

For reference: Maxwell (sm_50/52), Pascal (sm_60/61), Volta (sm_70), Turing (sm_75), Ampere (sm_80/86), Ada Lovelace (sm_89), Hopper (sm_90).

Compilation Workflow

Write Your Source Code

Create a file with .cu extension containing your CUDA program with host and device code.

Compile with nvcc

Run the nvcc compiler to transform your CUDA source into a binary executable.

Run Your Program

Execute the compiled program. It will automatically utilize your GPU when kernels are launched.

Full compilation example 
# Check if nvcc is installed and available
which nvcc
nvcc --version

# Compile your program with optimization and debug info
nvcc -O3 -arch=sm_75 -lineinfo my_kernel.cu -o my_kernel

# Run your CUDA program
./my_kernel

Using CMake with CUDA

For larger projects, you'll want to use a build system like CMake. Modern CMake (3.8+) has built-in CUDA support:

CMakeLists.txt 
cmake_minimum_required(VERSION 3.18)
project(my_cuda_project LANGUAGES CXX CUDA)

# Enable CUDA runtime
enable_language(CUDA)

# Set CUDA architecture
set(CMAKE_CUDA_ARCHITECTURES 75)

# Create executable
add_executable(my_program main.cu)

📝 Writing Your First CUDA Program

Let's walk through creating a complete CUDA program. We'll start with the classic "Hello GPU" example and build up to a working vector addition program that actually demonstrates GPU parallelism.

The Minimal CUDA Program

Every CUDA program has the same basic structure: host code (running on CPU) that allocates memory, launches kernels, and handles the results. Here's a minimal example:

hello_cuda.cu 
#include 

// This is the CUDA kernel - it runs on the GPU
__global__ void helloFromGPU() {
    printf("Hello from GPU thread %d\n", threadIdx.x);
}

int main() {
    printf("Hello from CPU!\n");
    
    // Launch the kernel with 1 block of 4 threads
    // <<>>
    helloFromGPU<<<1, 4>>>();
    
    // Wait for GPU to finish before CPU exits
    cudaDeviceSynchronize();
    
    return 0;
}

🔑 Key Concepts

Notice the triple angle bracket syntax <<<1, 4>>>. This is the kernel launch configuration that specifies how many blocks and how many threads per block to create. threadIdx.x is a built-in variable that gives each thread its unique ID.

First, let's look at the sequential C version so we can compare:

vector_add_cpu.c 
#include 
#include 

// Sequential vector addition - runs on CPU
void vectorAdd(const float *A, const float *B, float *C, int n) {
    for (int i = 0; i < n; i++) {
        C[i] = A[i] + B[i];
    }
}

int main() {
    int n = 1000000;  // 1 million elements
    size_t size = n * sizeof(float);
    
    // Allocate memory on host
    float *h_A = (float*)malloc(size);
    float *h_B = (float*)malloc(size);
    float *h_C = (float*)malloc(size);
    
    // Initialize arrays
    for (int i = 0; i < n; i++) {
        h_A[i] = 1.0f;
        h_B[i] = 2.0f;
    }
    
    // Perform vector addition on CPU
    vectorAdd(h_A, h_B, h_C, n);
    
    // Verify result
    printf("C[0] = %f\n", h_C[0]);
    printf("C[%d] = %f\n", n-1, h_C[n-1]);
    
    // Free memory
    free(h_A);
    free(h_B);
    free(h_C);
    
    return 0;
}

Now let's convert this to CUDA. The key changes are:

  1. Add __global__ qualifier to make a function a kernel
  2. Replace malloc() with cudaMallocManaged()
  3. Replace free() with cudaFree()
  4. Launch the kernel with execution configuration
  5. Add cudaDeviceSynchronize() to wait for GPU
vector_add_gpu.cu 
#include 
#include 

// CUDA kernel - runs on GPU
// Each thread handles one element
__global__
void vectorAdd(const float *A, const float *B, float *C, int n) {
    // Calculate global thread index
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    
    // Grid-stride loop for large arrays
    int stride = blockDim.x * gridDim.x;
    for (; i < n; i += stride) {
        C[i] = A[i] + B[i];
    }
}

int main() {
    int n = 1000000;  // 1 million elements
    size_t size = n * sizeof(float);
    
    // Allocate Unified Memory accessible from CPU and GPU
    float *d_A, *d_B, *d_C;
    cudaMallocManaged(&d_A, size);
    cudaMallocManaged(&d_B, size);
    cudaMallocManaged(&d_C, size);
    
    // Initialize arrays on the host
    for (int i = 0; i < n; i++) {
        d_A[i] = 1.0f;
        d_B[i] = 2.0f;
    }
    
    // Launch kernel on GPU
    // 256 threads per block, enough blocks to cover all elements
    int blockSize = 256;
    int numBlocks = (n + blockSize - 1) / blockSize;
    vectorAdd<<>>(d_A, d_B, d_C, n);
    
    // Wait for GPU to finish
    cudaDeviceSynchronize();
    
    // Verify result on CPU
    printf("C[0] = %f\n", d_C[0]);
    printf("C[%d] = %f\n", n-1, d_C[n-1]);
    
    // Check for errors
    float maxError = 0.0f;
    for (int i = 0; i < n; i++) {
        maxError = fmax(maxError, fabs(d_C[i] - 3.0f));
    }
    printf("Max error: %f\n", maxError);
    
    // Free GPU memory
    cudaFree(d_A);
    cudaFree(d_B);
    cudaFree(d_C);
    
    return 0;
}

Understanding the Execution Configuration

The <<<numBlocks, blockSize>>> syntax is crucial to understand. It determines how many parallel threads will execute your kernel:

Parameter Example Values Meaning
numBlocks 1, 64, 4096 How many blocks of threads to create
blockSize 32, 64, 128, 256, 1024 Threads per block (must be multiple of 32)

The total number of threads is numBlocks × blockSize. For our 1 million element array with 256 threads per block, we need approximately 3907 blocks.

The Grid-Stride Loop Pattern

Notice in the kernel we use a grid-stride loop:

Grid-stride loop pattern 
int i = blockIdx.x * blockDim.x + threadIdx.x;
int stride = blockDim.x * gridDim.x;
for (; i < n; i += stride) {
    // Process element i
}

This pattern is essential because:

  • It allows processing arrays larger than the total thread count
  • Each thread processes multiple elements if needed
  • It's the standard pattern for CUDA kernels that process arrays

✓ Compiling and Running

Save the CUDA code to vector_add_gpu.cu and compile with: nvcc vector_add_gpu.cu -o vector_add_gpu. Then run with ./vector_add_gpu.

🔄 How CUDA Programs Execute

Understanding the execution model is crucial for writing efficient CUDA programs. This section explains the complete lifecycle of a CUDA program and how it differs from traditional C programs.

The Two-Language Reality

Every CUDA program actually consists of two programs running on two different processors:

Host Code (CPU)

Runs on the CPU in regular C. Handles:

  • Memory allocation
  • Data transfers to/from GPU
  • Kernel launches
  • Computing the "main" logic

Device Code (GPU)

Runs on the GPU via CUDA kernels. Handles:

  • Parallel computations
  • Data-parallel operations
  • High-throughput calculations

⚠️ The Bandwidth Reality

PCIe transfers between CPU and GPU are slow (10-20 GB/s) compared to GPU memory bandwidth (500-1000 GB/s). This is why we prefer to keep data on the GPU and minimize transfers. Unified Memory helps, but explicit prefetching is still important for performance.

Asynchronous Execution

Kernel launches are asynchronous—the CPU continues executing while the GPU works. This means you must explicitly synchronize when needed:

Synchronization patterns 
// Kernel launch returns immediately (non-blocking)
myKernel<<>>(params);

// CPU continues executing here while GPU runs...

// This blocks until all GPU operations complete
cudaDeviceSynchronize();

// NOW we can safely read GPU results from CPU

// Check for any errors that occurred during execution
cudaError_t error = cudaGetLastError();
if (error != cudaSuccess) {
    printf("CUDA Error: %s\n", cudaGetErrorString(error));
}

🧠 Mental Model for CUDA Success

Writing successful CUDA programs requires thinking differently about computation. This section provides the mental frameworks and heuristics that will help you write correct and performant GPU code.

Think Data-Parallel, Not Sequential

The most fundamental shift is from thinking "how do I do this step-by-step" to "how do I do this to every element at once." Consider this analogy:

A regular C program is like one person painting a fence. A CUDA program is like 1000 people each painting their own section of the fence simultaneously. The speedup comes from the parallelism, but you must be able to divide the work into independent pieces.

The SIMT Model

CUDA threads execute in a Single Instruction, Multiple Thread (SIMT) fashion. This means:

  • All threads in a warp (32 threads) execute the same instruction at the same time
  • If threads branch differently (if/else), they serialize—that's called warp divergence
  • Threads can proceed independently when they have the same execution path

💡 Warp Divergence Example

Avoid patterns like if (threadIdx.x % 2 == 0) { ... } else { ... } because it causes divergence. All threads in a warp should follow the same execution path when possible.

The Hierarchy of Considerations

When writing CUDA code, think in this order of importance:

1️⃣

Correctness First

Does my code produce the right answer? Start simple, verify correctness before optimizing.

2️⃣

Memory Access

Are threads accessing memory efficiently? Coalesced access to global memory is crucial.

3️⃣

Occupancy

Are we using enough threads to keep all GPU cores busy? Aim for high occupancy.

4️⃣

Shared Memory

Can we use shared memory to reduce global memory traffic? It's faster but requires synchronization.

Memory Access Patterns

How your threads access memory determines performance. CUDA hardware is optimized for coalesced access—when consecutive threads access consecutive memory locations:

✓ Good: Coalesced access 
// Each thread accesses its own element
int i = blockIdx.x * blockDim.x + threadIdx.x;
result[i] = data[i] + data[i + 1];  // Consecutive threads, consecutive memory
✗ Bad: Strided access 
// Each thread accesses elements far apart
int i = threadIdx.x;
result[i] = data[i * stride];  // Thread 0 accesses data[0], thread 1 accesses data[1000]

The Golden Rules

  1. Keep data on the GPU as long as possible. Minimize transfers between CPU and GPU memory.
  2. Use 256 or 512 threads per block. This usually gives good occupancy and works well with the hardware warp size of 32.
  3. Prefer simple arithmetic over complex control flow. GPUs have massive parallelism but limited branch prediction.
  4. Profile before optimizing. Use nvprof, Nsight Systems, or Nsight Compute to find actual bottlenecks.
  5. Start with unified memory. It's simpler than manual memory management, and let the profiler guide you if you need manual control.

Debugging Strategies

Debugging GPU code is different from debugging CPU code:

Issue Strategy
Logic errors Print from kernels with printf(), verify with small arrays
Race conditions Avoid by design; use atomic operations only when necessary
Memory errors Use cuda-memcheck for validation
Performance issues Profile with Nsight Systems, focus on memory bandwidth utilization

💡 Quick Debug Tip

Start with a tiny array (10-100 elements) and print out all values. Once you verify correctness, scale up to your full dataset. This catches bugs early before they become obscure.

When NOT to Use CUDA

CUDA isn't always the right tool. Avoid GPU acceleration when:

  • The problem is inherently sequential (hard to parallelize)
  • The dataset is too small (overhead exceeds benefit)
  • You're doing complex branching with no data parallelism
  • Memory access patterns are highly irregular

⚠️ Amdahl's Law

The speedup from parallelization is limited by the sequential portion of your code. If only 10% of your program can be parallelized, even perfect parallelization gives only 10x speedup maximum. Choose problems with high parallelism potential.

💡 Beginner Project Ideas

Here are progressively challenging project ideas to practice your CUDA skills. Start with the simpler ones and work your way up as you gain confidence.

Level 1: Fundamentals (Beginner)

1. Vector Operations Suite

Beginner

Implement a collection of basic vector operations to get comfortable with CUDA syntax and memory management.

  • Vector addition, subtraction, multiplication
  • Scalar operations (multiply by constant)
  • Dot product
  • Vector magnitude

Learning goals: Kernel syntax, thread indexing, memory allocation, error handling.

2. Histogram Computation

Beginner

Count the occurrences of values in an array, producing a frequency distribution.

  • Given an array of random integers, count how many fall into each bin
  • Start with simple bin counting, then optimize with atomic operations
  • Visualize the histogram on the CPU

Learning goals: Atomic operations, shared memory basics, handling write conflicts.

3. Matrix Multiplication

Beginner

Implement standard matrix multiplication and compare CPU vs GPU performance.

  • Simple O(n³) implementation first
  • Use 2D thread blocks for natural matrix indexing
  • Benchmark against CPU implementation

Learning goals: 2D thread organization, global memory access patterns, performance comparison.

Level 2: Intermediate Projects

4. Parallel Reduction (Sum, Max, Min)

Intermediate

Implement efficient parallel reduction algorithms—the building blocks of many GPU applications.

  • Implement multiple reduction strategies
  • Use shared memory for thread-block local reduction
  • Compare naive vs optimized implementations
  • Profile to see the performance difference

Learning goals: Shared memory, warp-level primitives, algorithm optimization.

5. Image Processing Filters

Intermediate

Implement common image processing operations using GPU parallelism.

  • Blur (box blur, Gaussian blur)
  • Edge detection (Sobel, Laplacian)
  • Histogram equalization
  • Load and save images using stb_image library

Learning goals: 2D data structures, boundary handling, texture memory (optional advanced topic).

6. Prefix Sum (Scan)

Intermediate

Implement parallel prefix sum, one of the most important parallel algorithms.

  • Implement Hillis-Steele algorithm
  • Implement more efficient Blelloch algorithm
  • Use prefix sum to accelerate other algorithms

Learning goals: Work-efficiency, algorithmic thinking, using scan for other problems.

Level 3: Advanced Explorations

7. Sparse Matrix-Vector Multiplication

Advanced

Multiply a sparse matrix by a vector—common in scientific computing and machine learning.

  • Use CSR (Compressed Sparse Row) format
  • Handle load balancing for irregular sparsity
  • Compare with MKL/cuSPARSE libraries

Learning goals: Irregular data structures, memory efficiency, library interop.

8. Monte Carlo Pi Estimation

Advanced

Use random sampling to estimate the value of π with GPU parallelism.

  • Generate random points in a unit square
  • Count points inside quarter circle
  • Parallel random number generation
  • Statistical analysis of accuracy vs sample count

Learning goals: Random number generation on GPU, statistical methods, atomic operations.

9. N-Body Simulation

Advanced

Simulate the gravitational interaction of N particles—O(n²) but perfect for GPU parallelization.

  • Calculate pairwise forces between all particles
  • Update velocities and positions
  • Visualize with a simple graphics library
  • Optimize with shared memory tiling

Learning goals: O(n²) parallelism, shared memory optimization, physics simulation.

Project Checklist

Use this checklist to ensure you're following best practices in each project:

  • Start with a working CPU version for correctness verification
  • Implement the simplest CUDA version first
  • Verify CUDA output matches CPU output
  • Profile and identify bottlenecks
  • Add error checking to all CUDA API calls
  • Benchmark and compare CPU vs GPU performance
  • Document your findings and optimization steps
