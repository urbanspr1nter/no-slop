Parallel Thinking: A CUDA Tutorial

From Sequential C to Massively Parallel GPU Programming

1. What is CUDA?

If you are coming from a background in standard C programming on Linux, you are likely used to thinking about your code as a sequence of instructions executed by a highly intelligent, yet solitary, worker: the CPU. Your CPU is designed to minimize latency. It wants to execute a single thread of instructions as fast as possible, using complex branch prediction, deep pipelines, and massive caches to ensure that the next instruction is always ready.

CUDA (Compute Unified Device Architecture) is a parallel computing platform and programming model created by NVIDIA. It allows you to offload computationally intensive tasks from the CPU to the GPU (Graphics Processing Unit). Unlike the CPU, the GPU is designed for throughput. Instead of one incredibly fast worker, imagine having thousands of relatively simple workers. While a single worker might be slower than a CPU core at any given task, their collective power when performing the same task on different pieces of data is orders of magnitude greater.

In C, you might write a loop to process an array. In CUDA, you don't write the loop; you write the logic for what a single element should do, and you tell the GPU to launch thousands of copies of that logic simultaneously.

2. Compiling CUDA Programs

You cannot use standard `gcc` to compile CUDA code because `gcc` does not understand the special syntax used to define GPU kernels (the functions that run on the GPU). Instead, NVIDIA provides a compiler called nvcc (NVIDIA CUDA Compiler).

A CUDA source file typically uses the .cu extension. When you run nvcc, it performs a split-compilation process. It identifies which parts of your code are meant for the CPU (the "host") and which parts are meant for the GPU (the "device"). It compiles the host code using a standard compiler like GCC and compiles the device code into PTX (Parallel Thread Execution) instructions, which the GPU hardware eventually executes.

A typical compilation command on a Linux system looks like this:

nvcc -o my_program my_program.cu

If you are using external libraries or specific architectures, you might add flags such as -arch=sm_80 (to target a specific GPU architecture like Ampere) or -I/path/to/includes. For a beginner, the simple command above is often sufficient to get started.

3. Your First CUDA Program: Vector Addition

To understand the syntax, let's look at the classic "Hello World" of parallel computing: adding two vectors. In standard C, you would loop through an array. In CUDA, we define a kernel.

#include <stdio.h>
#include <cuda_runtime.h>

// The Kernel: This runs on the GPU
// __global__ tells the compiler this function is called from CPU and runs on GPU
__global__ void vectorAdd(const float* A, const float* B, float* C, int numElements) {
    // Calculate the unique index for this specific thread
    int i = blockDim.x * blockIdx.x + threadIdx.x;

    // Check if we are within the array bounds
    if (i < numElements) {
        C[i] = A[i] + B[i];
    }
}

int main() {
    int numElements = 50000;
    size_t size = numElements * sizeof(float);

    // 1. Allocate memory on the Host (CPU)
    float *h_A = (float *)malloc(size);
    float *h_B = (float *)malloc(size);
    float *h_C = (float *)malloc(size);

    // Initialize host arrays
    for (int i = 0; i < numElements; ++i) {
        h_A[i] = 1.0f;
        h_B[i] = 2.0f;
    }

    // 2. Allocate memory on the Device (GPU)
    float *d_A, *d_B, *d_C;
    cudaMalloc(&d_A, size);
    cudaMalloc(&d_B, size);
    cudaMalloc(&d_C, size);

    // 3. Copy data from Host to Device
    cudaMemcpy(d_A, h_A, size, cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_B, size, cudaMemcpyHostToDevice);

    // 4. Launch the Kernel
    // We define a block size (threads per block) and calculate the number of blocks needed
    int threadsPerBlock = 256;
    int blocksPerGrid = (numElements + threadsPerBlock - 1) / threadsPerBlock;

    // The <<<...>>> syntax is the execution configuration
    vectorAdd <<<blocksPerGrid, threadsPerBlock>>>(d_A, d_B, d_C, numElements);

    // 5. Copy the result back from Device to Host
    cudaMemcpy(h_C, d_C, size, cudaMemcpyDeviceToHost);

    // Verify results
    for (int i = 0; i < 10; ++i) {
        printf("%f + %f = %f\n", h_A[i], h_B[i], h_C[i]);
    }

    // 6. Cleanup
    cudaFree(d_A);
    cudaFree(d_B);
    cudaFree(d_C);
    free(h_A);
    free(h_B);
    free(h_C);

    return 0;
}
Key Concept: Notice the memory management. In C, malloc is enough. In CUDA, you must explicitly allocate memory on the GPU using cudaMalloc and explicitly move data between the CPU and GPU using cudaMemcpy. The GPU cannot see your CPU's RAM directly.

4. The Execution Model: Grids, Blocks, and Threads

One of the most confusing parts for C programmers is the hierarchy of execution. In C, you have a single program counter. In CUDA, you have a massive hierarchy of threads.

When you launch a kernel using the <<<blocks, threads>>> syntax, you are defining a Grid. This grid is made up of several Blocks, and each block is made up of several Threads. This hierarchy is designed to allow the hardware to scale. If you have a small GPU, it runs a few blocks. If you have a massive GPU, it runs many more.

To know which part of the data a thread should work on, each thread uses its unique ID. This ID is calculated using three built-in variables:

  • threadIdx.x: The index of the thread within its block.
  • blockIdx.x: The index of the block within the grid.
  • blockDim.x: The number of threads in a single block.

The formula int i = blockIdx.x * blockDim.x + threadIdx.x; is the standard way to map a 1D thread index to a global index in your array.

5. Mental Models for Success

To be successful in CUDA, you must fundamentally change how you approach problem-solving. Here are three pillars of the CUDA mindset:

First, think in Data Parallelism, not Instruction Parallelism. In C, you think: "First do A, then do B, then loop through C." In CUDA, you think: "I have 1,000,000 pieces of data; what is the transformation that applies to one piece of data?" If your algorithm is inherently sequential (like a linked list traversal), CUDA will not help you. If it is independent (like adding two arrays), CUDA will shine.

Second, respect the Memory Wall. The bottleneck in GPU programming is rarely the math; it is almost always the movement of data. Moving data from the CPU to the GPU via the PCIe bus is extremely slow compared to the speed of the GPU cores. A successful CUDA programmer spends more time thinking about how to minimize cudaMemcpy calls and how to ensure "coalesced" memory access (reading contiguous chunks of memory) than they do about optimizing the math itself.

Third, embrace the "Compute-to-Communication" ratio. Because moving data is expensive, you want your kernels to do as much work as possible once the data is on the GPU. If you move data to the GPU, perform one addition, and move it back, you have wasted your time. If you move data to the GPU, perform 1,000 different operations, and then move it back, you have used the GPU effectively.

6. Beginning Project Ideas

Once you have mastered vector addition, don't just stop. Try these projects in order of increasing difficulty:

1. Dot Product: This is slightly harder than vector addition because it requires a "reduction" (summing all the results into a single value). This will teach you about synchronization and shared memory.

2. Image Grayscale Conversion: Treat an image as a large 2D array of pixels. Each thread can be responsible for calculating the brightness of a single pixel. This introduces you to 2D grids and blocks.

3. Matrix Multiplication: The cornerstone of Deep Learning. Implementing a naive matrix multiplication is a great way to understand 2D indexing, and then trying to optimize it using Shared Memory is where the real learning happens.
